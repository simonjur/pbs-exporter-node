# 2. Runtime & build

> Part of the [PBS Exporter specification](../SPEC.md). See the index for the
> spec-driven-development workflow and the meaning of **[offline-ok]** / **[needs-pbs]**.

| ID | Requirement | Verify |
|----|-------------|--------|
| REQ-RT-1 | Runs on Node.js >= 24 directly from TypeScript (native type stripping); no separate build step required to run. | `node src/run.ts --version` prints a version line and exits 0. **[offline-ok]** |
| REQ-RT-2 | Type-checks cleanly. | `npm run lint:ts` exits 0. **[offline-ok]** |
| REQ-RT-3 | Core runtime dependencies are `prom-client`, `undici`, `commander`, `parse-duration`, and `winston` (logging); the [status UI](status-ui.md) additionally depends on `vue` and `vuetify`, whose browser builds are assembled into `public/` by `npm run build:fe` (see `REQ-UI-6`). The TypeScript backend itself still runs without a compile step. | Inspect `package.json` `dependencies` and the `build:fe` script. **[offline-ok]** |
| REQ-RT-4 | Config-building logic lives in [`src/config.ts`](../src/config.ts) as a pure `loadConfig(opts, env)` function (no commander, no `process.argv`), so it can be unit-tested in isolation (see [Testing](testing.md), `REQ-TEST-*`). All commander/CLI parsing lives in the entrypoint [`src/run.ts`](../src/run.ts), which passes the parsed options to `loadConfig`. | Inspect the module structure: `config.ts` exports `loadConfig`/`CliOptions` and imports no `commander`; `run.ts` imports `commander` and calls `loadConfig`. **[offline-ok]** |
| REQ-RT-6 | All `.ts` files and the status-UI script [`src/web/app.js`](../src/web/app.js) lint cleanly under ESLint 10 (flat config in [`eslint.config.mjs`](../eslint.config.mjs), using `@eslint/js` + `typescript-eslint` recommended rules). Unused variables/arguments/imports are an **error** (`@typescript-eslint/no-unused-vars` for `.ts`, `no-unused-vars` for `app.js`), with a `_`-prefix escape hatch for intentionally-unused bindings. | `npm run lint:eslint` exits 0. **[offline-ok]** |
| REQ-RT-7 | All `.ts` files conform to Prettier formatting. | `npm run lint:prettier` exits 0 (`npm run format` auto-fixes). **[offline-ok]** |
| REQ-RT-8 | [`src/run.ts`](../src/run.ts) is the thin process entrypoint (CLI parsing + bootstrap) and calls `main(config)` exported from [`src/main.ts`](../src/main.ts), which wires the collaborators and starts the server. The PBS API client and HTTP layer live in dedicated modules ([`src/exporter.ts`](../src/exporter.ts), [`src/server.ts`](../src/server.ts)) so they can be unit-tested in isolation (see [Testing](testing.md), `REQ-TEST-*`). | Inspect the module structure: `run.ts` parses argv and calls `main`; `main.ts` exports `main(config)` and wires collaborators it imports from `config.ts`, `exporter.ts`, and `server.ts`. **[offline-ok]** |
